== Executive Summary

=== Context

Your organisation is an established European grocery and commodity webshop — 15 years on the market, ~1M daily visitors, 2M mobile app installs. The goal is to increase annual turnover by 20%. Today, customer data is scattered across isolated systems: the WordPress e-commerce platform, the mobile application, and wholesaler product feeds each operate independently, with no unified view of purchasing behaviour.

This document proposes a unified data pipeline that consolidates customer, product, and wholesaler data into a single analytical layer — and uses it to deliver personalised product suggestions via email and mobile push notifications. The platform is designed to be owned and operated by your existing team, without requiring a dedicated data function.

=== Scope Boundary

This engagement delivers: unified data pipeline (WordPress, mobile app, wholesalers), Snowflake analytical platform, customer segmentation and recommendation engine, email campaign delivery via AWS SES, mobile push notifications, and a self-service analytics dashboard.

This engagement explicitly *does not* include: WordPress migration or re-platforming, new feature development for the mobile application, EDI or real-time API integration with wholesalers, real-time personalisation, or recruitment of a permanent data team. Any change to this scope is subject to a formal change request and re-estimation.

=== Target State

The enterprise has a unified data platform that continuously consolidates customer, product, and wholesaler data from all sources into a single analytical store. A personalised suggestion engine analyses buying patterns and matches customers with relevant products and deals, delivering timely recommendations via email and mobile push. The marketing team monitors campaign performance through self-service dashboards, and the platform is designed for the internal team to gradually take ownership with minimal external dependency.

=== Business Goal Translation

Your stated goal is "increase turnover by 20%." In technical terms this means:

* **Unify** all customer data islands into a single source of truth
* **Profile** customers based on purchase history, browsing behaviour, and preferences
* **Recommend** products with the highest conversion probability per customer segment
* **Deliver** recommendations through email (AWS SES) and mobile push (2M installs)
* **Measure** uplift via campaign tracking and A/B testing in the analytics dashboard

=== What It Delivers

* Unified analytical data store (Snowflake) consolidating WordPress, mobile, and wholesaler data
* Automated ETL pipeline with data quality validation and email address verification
* ML-based customer segmentation and personalised product recommendation engine
* Personalised email campaign delivery via **AWS SES** (already in use)
* Mobile push notification capability (Phase 2 — 2M installs makes this a **primary channel**)
* Self-service analytics dashboard for marketing team (campaign performance, KPIs)
* GDPR-compliant data processing with consent management and customer deletion support

== Architecture Decisions

=== Decision 1: Data Platform — Snowflake + dbt + Airflow

**Decision:** Snowflake is the analytical data platform. dbt handles SQL transformations. Airflow (AWS MWAA) handles orchestration.

**Rationale:** Your team currently has no dedicated data engineering expertise, and there are no plans to hire. The platform must be SQL-first, managed, and operable by non-specialists. Snowflake auto-scales compute, suspends when idle (cost control), and runs in EU AWS regions (GDPR compliance). dbt provides version-controlled, testable transformations with a low learning curve. MWAA removes the operational burden of self-managing Airflow.

[cols="2,3,3"]
|===
|Alternative |Why Considered |Why Not Chosen

|Databricks (Data Lakehouse)
|Raised during discovery; better for custom ML model training; supports real-time streaming
|Over-engineered for Phase 1 scope; requires Spark expertise the team does not have; $800K+ cost exceeds confirmed $500K budget

|Custom ETL + PostgreSQL / BigQuery
|Lowest licence cost; no vendor lock-in
|High operational burden contradicts the non-technical team profile; time-to-value significantly longer; ops cost offsets licence savings
|===

**Consequences:** Vendor lock-in to Snowflake SQL extensions and Snowpark. If advanced custom ML model training becomes necessary (Phase 4+), Databricks or SageMaker can be layered in without replacing Snowflake. Snowflake skills are widely available in the hiring market.

---

=== Decision 2: Data Ingestion Strategy

**Decision:** Each source uses the lowest-complexity ingestion pattern that meets the analytical need.

[cols="1,2,3"]
|===
|Source |Chosen Approach |Rationale

|WordPress DB (900GB, heavily customised MySQL, on-prem)
a|Two options depending on IT capability — confirmed in Phase 1:

* **Option A: Nightly full export** (mysqldump → S3). Simple cron job, no DBA skills required. Full snapshot, not delta — sufficient for weekly recommendations. Storage: ~9–54GB/night compressed.
* **Option B: MySQL read replica** (on-prem replica → delta SQL extraction → S3). Requires DBA to configure replication. Cleaner long-term — only changed rows transferred, smaller loads, standard incremental pattern.

Default to Option A; upgrade to Option B if IT team can support replication or if daily refresh is required.
|Both options avoid any connection to the live production database.

|Mobile App Vendor (2M installs, external backend)
|Primary: vendor-managed S3 export. Fallback: direct read-only DB access. API for delta sync only once auth and rate limits are confirmed.
|Vendor-managed S3 avoids rate limits and schema coupling. DB access confirmed available on request. API paginating 2M records weekly is high-risk — restricted to delta sync only.

|Wholesaler Feeds (no standard format — CSV, Excel, XML)
|S3 file landing zone; Snowflake VARIANT columns for schema-on-read; per-supplier dbt models for normalisation.
|No standard format makes a unified schema impossible at ingestion. Schema-on-read defers normalisation to dbt where it can be version-controlled and tested. Start with 1–2 suppliers to validate the pattern.

|Internal Events (email delivery, opens, clicks)
|AWS SES webhooks → S3 → dbt → `mart.email_events`.
|Closes the recommendation feedback loop — open/click/convert data becomes the training signal for the Phase 3 ML model.
|===

**Convergence point:** All sources → S3 staging → Snowflake landing zone → dbt models → analytical mart.

---

=== Decision 3: Dashboard & BI Tooling — Metabase

**Decision:** Metabase (self-hosted on ECS) as the analytics dashboard for the marketing team.

**Rationale:** The marketing team has no SQL expertise. Metabase provides a point-and-click query builder, pre-built chart types, and scheduled email reports — no SQL required for dashboard consumers. Self-hosted on ECS eliminates the $500/month Metabase Cloud licence while keeping full control. Snowflake native connector is available and well-supported.

[cols="2,3"]
|===
|Alternative |Why Not Chosen

|Apache Superset
|Better for SQL-fluent analyst teams; steeper UI for non-technical users; requires more setup and maintenance

|Custom dashboard (React + charting library)
|Expensive to build and maintain; no benefit over an off-the-shelf tool for standard marketing KPIs
|===

**Consequences:** Marketing team can self-serve campaign performance data without engineer involvement. Metabase version upgrades become a maintenance responsibility (low frequency, low risk). If analytical needs outgrow Metabase, migration to Superset or Tableau is straightforward — data stays in Snowflake.

---

=== Decision 4: Recommendation Algorithm — Collaborative Filtering with Rule-Based Baseline

**Decision:** Phase 2 launches with a rule-based baseline (purchase frequency + category affinity). Phase 3 introduces item-based collaborative filtering trained on order history.

**Rationale:** A rule-based baseline can be built entirely in SQL on Snowflake without a data scientist, delivers measurable uplift in Phase 2, and establishes the feedback loop (open rates, click rates, conversions) needed to validate and improve the ML model. Item-based collaborative filtering is the industry-standard starting point for grocery e-commerce — it handles the cold-start problem better than user-based CF for customers with sparse history.

[cols="2,1,2"]
|===
|Approach |Phase |Minimum Data Requirement

|Rule-based segmentation (purchase frequency, category affinity, recency)
|Phase 2
|≥3 months of order history per customer; ≥2 orders to classify as active

|Item-based collaborative filtering (customers who bought X also bought Y)
|Phase 3
|≥50K customers with ≥2 orders; item co-occurrence matrix rebuilt weekly in Snowflake

|Neural collaborative filtering / deep learning
|Future (Phase 4+)
|≥500K active customers; requires Databricks or SageMaker — separate decision
|===

**Output format:** Ranked list of up to 10 product IDs per `customer_id`, refreshed weekly, stored in the `mart.recommendations` Snowflake table. Cold-start customers (no purchase history) receive category bestsellers based on regional popularity.

**Consequences:** Phase 2 email campaigns can launch without a trained ML model, reducing the dependency on a single ML engineer. The feedback data collected in Phase 2 (open, click, convert events) becomes the training signal for the Phase 3 model.
